# FSHD 前端重设计要求与提示词

## 1. 项目目标

在不改变现有业务逻辑、接口协议、路由结构、状态流和数据录入方式的前提下，重新设计 `apps/mobile` 的前端界面。新的前端应从“功能页堆叠”升级为“FSHD 单病例专病 dashboard”，重点强化可视化表达，让用户和演示对象能够更快理解诊断依据、病程变化、肌力分布、影像受累模式和风险监测结论。

本次改版不是后端重构，不是流程改造，也不是字段重定义。允许重做信息架构、视觉系统、组件样式、图表表达、卡片层级和交互反馈；不允许改动 API 路径、请求返回格式、鉴权逻辑、页面跳转逻辑、表单提交含义和已有数据读写规则。

## 2. 当前项目边界

- 技术栈：Expo Router + React Native + TypeScript。
- 现有可用图形能力：`react-native-svg`、`react-native-chart-kit`、`expo-linear-gradient`。
- 主要页面结构：
  - `p-home`：首页总览。
  - `p-manage`：病程管理。
  - `p-archive`：档案与历史数据。
  - `p-report_detail`：单份报告详情。
  - 其他页如 `p-qna`、`p-community`、`p-settings` 保持逻辑不变，仅统一视觉体系。
- 现有数据来源主要来自：
  - `getMyPatientProfile()`
  - `getRiskSummary()`
  - `getMuscleInsight()`
  - `getSubmissionTimeline()`
  - `getPatientDocumentOcr()`

## 3. 重设计硬约束

- 不改变任何接口调用方式、字段命名、数据写入行为和鉴权流程。
- 不新增依赖于后端新字段的核心流程；缺失字段必须优雅降级，显示为空态、占位态或“暂无数据”。
- 不把可视化做成泛化 BI 看板，要保持“医学专病病例档案”的语境。
- 不把界面做成传统检验系统或密集表格墙，核心信息要明显前置。
- 不照搬当前紫色玻璃拟态风格，允许建立新的视觉系统，但要保留医疗场景的专业感、可信感和移动端可读性。
- 所有图表必须适配单病例、小样本、不完整时间点、相对时间描述和左右侧不对称数据。
- 所有页面必须先保证手机端体验，再兼容平板和 Web 预览。

## 4. 新前端的核心视觉方向

新的界面应该呈现“临床病例驾驶舱”而不是“普通健康打卡 App”。建议采用深色医学底板配低饱和强调色，风格关键词为：专业、克制、清晰、专病化、可汇报截图。页面需要有明显的信息分层，首屏先给结论，再给证据，再给展开细节。图表与卡片关系要紧密，避免把图单独塞进页面角落。

可使用的视觉原则：

- 首页第一屏必须在 3 秒内讲清“这是 FSHD1 病例，目前状态如何”。
- 每个可视化都要有一句可直接读出的结论摘要。
- 色彩优先表达语义，而不是装饰。风险、下降、异常、受累更重侧必须有稳定颜色规则。
- 图形尽量轻量，避免复杂 3D、强装饰发光、过多拟物。
- 要考虑截图汇报场景，单屏组件要足够完整。

## 5. 可视化内容要求

基于 PDF 里的 5 个 demo，新的前端至少要把以下模块纳入页面设计，并映射到现有路由中：

### 5.1 首页 `p-home`

- 角色：总览入口。
- 必须突出：
  - 患者身份与当前状态摘要。
  - 基因摘要卡。
  - 风险提示或近期状态结论。
  - 进入病程管理、档案、问答等模块的快捷入口。
- 建议加入：
  - “FSHD1 确诊”状态卡。
  - `4qA`、`EcoRI 20 kb`、`D4Z4 3/22` 的紧凑展示。
  - 最近一次关键变化或风险摘要。

### 5.2 病程管理 `p-manage`

- 角色：动态进展中枢。
- 必须承载：
  - 病程时间轴。
  - 肌力人体受累示意图。
  - 监测面板。
- 保留现有“病程管理”逻辑入口，但视觉上升级为专病分析页。
- 现有的 `DataComparisonFeature`、`RecordTimelineScreen` 可以重构为更统一的图形卡片，而不是零散卡块。

### 5.3 档案页 `p-archive`

- 角色：病例档案与历史记录中心。
- 需要从“数据列表”升级为“病例档案总册”。
- 可保留已有趋势图和雷达图，但整体层级应围绕：
  - 档案身份卡。
  - 提交/随访时间线。
  - 历史报告摘要。
  - 关键肌群趋势。

### 5.4 报告详情 `p-report_detail`

- 角色：单份报告的结构化阅读页。
- 除结构化字段和 AI 总结外，应加入更直观的视觉承载：
  - MRI 受累地图。
  - 人体分区说明与图例。
  - 报告来源与原文追溯。
- 点击字段后可展开来源，不改原有数据逻辑。

## 6. 五个重点 demo 的页面化要求

### Demo 1：基因摘要卡

- 用于首页最重要位置。
- 必须让用户一眼看懂为何确诊 FSHD1。
- 推荐图形：状态卡 + 数值对照条 + 结论标签。
- 必须保留 3 个核心数值：`4qA`、`EcoRI 20 kb`、`D4Z4 3/22`。

### Demo 2：病程时间轴

- 适配 `p-manage` 的主模块之一。
- 移动端可优先使用纵向时间轴。
- 要支持相对时间，如“5 年前”“10 个月前”，不能强制依赖精确日期。
- 时间轴节点只保留关键事件，不堆满所有检查。

### Demo 3：肌力热图

- 适合作为 `p-manage` 的核心主体。
- 必须突出：
  - 选择性受累。
  - 左右差异。
  - 近端/轴性为主。
- 不采用表格热图作为主表达，主视觉必须是正面/背面人体示意图。
- 通过人体不同区域的填色、描边、强弱层级或局部高亮来表示肌群受累程度。
- 左右差异必须直接映射在人体左右两侧，而不是放在表格列里比较。
- 原始分值可以作为点击后展开信息或悬浮说明存在，但不能抢占主画面。

### Demo 4：MRI 肌群受累地图

- 适合在 `p-report_detail` 或 `p-archive` 中作为重点分析模块。
- 推荐形式：简化人体前后示意图，作为绝对主视觉。
- 首版不要求高精度解剖插画，但必须有清晰左右侧和部位表达。
- 要支持脂肪浸润、炎性改变、萎缩并存的非互斥表达。
- 不要把 MRI 受累信息做成表格或矩阵作为主界面主体。
- 可以在人体图旁边配少量说明标签、图例和结论摘要，但不要回退成数据表阅读模式。

### Demo 5：呼吸 / 心脏 / 肌酶监测面板

- 适合作为 `p-manage` 的风险总览模块。
- 用三列或三段式卡片布局展示：
  - 呼吸。
  - 心脏。
  - 肌酶。
- 强调“一眼判断是否存在风险”，不要堆全部指标。
- 单位不同的实验室指标不强行共用一个坐标系。

## 7. 交互要求

- 点击图表节点、字段、标签时，可展开来源说明或原始文本片段，但不改变当前接口行为。
- 所有可视化都必须有空态、加载态、错误态。
- 页面动效要服务于信息层级，例如首屏卡片渐入、图表逐步显现、节点展开，而不是无意义炫技。
- 保持移动端触控友好，按钮与点击区足够大。

## 8. 输出给设计/生成模型时的验收标准

- 不看原始 PDF，也能看懂 FSHD1 的诊断依据、病程、表型和监测重点。
- 首页首屏能快速建立“病例是什么、当前怎么样”的认知。
- 病程页和档案页不再只是数据堆叠，而是有叙事顺序。
- 图表不是装饰，必须直接帮助解释医学信息。
- 改版后仍然能直接复用现有页面逻辑和接口。

## 9. 可直接使用的提示词

你现在要为一个已经存在的 Expo Router + React Native + TypeScript 医疗项目重设计前端界面。这个项目的业务逻辑、API 接口、鉴权流程、路由结构、数据字段和页面功能暂时都不能改变，你只能重做前端的信息架构、视觉系统、组件布局、图表表达和交互层次。项目是一个 FSHD 患者单病例管理应用，当前已有首页、病程管理、档案页、报告详情页、问答页、社区页和设置页。新的设计目标不是做一个普通健康 App，而是做成一个“FSHD 单病例专病 dashboard”，让用户或医生在很短时间内看懂诊断依据、病程进展、肌力分布、MRI 受累模式和呼吸/心脏/肌酶监测结果。

请严格遵守以下要求：第一，不改变任何现有逻辑，不新增依赖后端新字段的关键流程，不修改 API 请求与返回结构，不改变导航关系和页面职责；第二，所有可视化必须适配移动端，优先服务手机屏幕，同时兼容平板和 Web 预览；第三，视觉风格要专业、克制、专病化、适合截图汇报，不要做成传统体检系统，也不要做成泛科技玻璃拟态页面；第四，必须围绕 5 个重点 demo 组织页面，即基因摘要卡、病程时间轴、肌力热图、MRI 肌群受累地图、呼吸/心脏/肌酶监测面板；第五，Demo 3 和 Demo 4 不要用表格、矩阵或列表作为主视觉，必须以人体示意图为核心表达方式，用正面/背面人体区域高亮来体现受累分布、左右差异和严重程度；第六，所有图表都要能处理单病例、小样本、字段不完整、相对时间和左右不对称问题，缺失数据时要显示优雅空态而不是伪造数据。

请按下面的页面映射来设计：`p-home` 作为病例总览首页，首屏突出 FSHD1 状态、基因摘要、风险结论和核心入口；`p-manage` 作为病程管理主页面，重点承载病程时间轴、肌力热图和监测面板；`p-archive` 作为病例档案总册，展示患者身份、历史提交、关键报告摘要和趋势信息；`p-report_detail` 作为单份报告的结构化阅读页，重点强化 MRI 受累地图、结构化字段、AI 总结和原始来源追溯。其余页面如问答、社区、设置只做视觉统一，不需要成为可视化重点页。

请输出一套高保真前端设计方案，内容至少包括：1）整体视觉方向和设计语言；2）颜色、字体、卡片、标签、状态色、图表色的设计 token；3）首页、病程管理页、档案页、报告详情页的页面结构；4）5 个 demo 分别采用什么图形形式，以及为什么适合 FSHD 单病例场景；5）重点说明 Demo 3 和 Demo 4 的人体示意图怎么设计，包括正面/背面切换、左右差异表达、图例系统和点击联动；6）空态、加载态、错误态和交互动效设计；7）哪些现有组件可以保留逻辑但重做表现；8）一套适合 React Native 落地的组件拆分建议。请直接给出可执行、可落地、具有明显视觉方向的设计方案，不要只写抽象原则。
