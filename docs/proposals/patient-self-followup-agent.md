# FSHD 患者自录与长期随访 Agent 文档

## 1. 文档目的

本文档用于指导后续实现 agent，对当前患者档案与随访模块进行产品和数据结构改造。改造目标不是把现有页面做得更复杂，而是把平台从“按数据类型堆表单”升级为“面向患者自录、长期随访、低负担、可理解的 FSHD 病程记录系统”。

本文档默认读者为实现 agent、产品设计 agent、前端 agent 与后端 agent。内容会同时覆盖产品定位、录入逻辑、数据模型、接口、页面信息架构、可视化规则和验收标准。

## 2. 产品定位

- 平台主要服务对象是 FSHD 患者本人，而不是医生端录入员。
- 平台核心价值是帮助患者持续记录病情变化、理解自己的病程、整理与医生沟通所需的信息。
- 平台不是临床评分系统替代品，不应要求患者完成高门槛、重专业解释的录入任务。
- 平台不应优先展示“原始医学数据堆叠”，而应优先展示“这段时间发生了什么变化”。

## 3. 当前问题

基于现有实现，当前系统存在以下偏差：

- 录入页仍按“报告 / 肌力 / 活动 / 用药”分块，用户需要自己理解哪些信息属于一次随访。
- 楼梯测试被写入活动日志，不利于后续做稳定趋势和比较。
- 肌力记录没有左右侧表达，不适合 FSHD 的不对称受累特征。
- 数据结构偏向通用档案系统，缺少患者最关心的症状变化、功能影响、跌倒、辅具、呼吸状态等长期随访信息。
- 可视化目前偏“通用图表 + 提交列表”，缺少患者视角的病程摘要与变化解释。
- 分位和同群体比较不应作为患者首页的核心信息，容易制造误导或额外焦虑。

## 4. 改造目标

本次改造必须实现以下目标：

- 让患者用最少的步骤完成一次随访记录。
- 让患者优先回答“和上次相比有没有变化”，而不是重复填写全部字段。
- 让系统支持 FSHD 特有的左右不对称、肩带/面部/下肢/呼吸等多域长期变化。
- 让首页与档案页优先展示“当前状态 + 与上次相比变化 + 关键证据”。
- 让上传报告成为辅助信息来源，而不是主录入方式。
- 让时间线表达“病程事件”，而不是仅表达“提交过哪些数据”。

## 5. 非目标

以下内容不在本轮核心范围内：

- 不做医生工作站式复杂评分面板。
- 不要求首期就支持全部神经肌肉专病量表。
- 不把患者端做成统计学比较系统，不以人群分位作为主要反馈。
- 不要求患者理解全部基因和影像专业术语。
- 不要求一次性重构所有历史页面，可按阶段渐进替换。

## 6. 设计原则

### 6.1 患者优先

- 所有录入流程必须以患者能否理解、能否坚持为第一判断标准。
- 能用通俗表达就不用专业缩写。
- 专业术语可以保留，但只能作为二级说明或展开内容。

### 6.2 变化优先

- 系统不应该每次都要求从头录完整档案。
- 每次随访默认围绕“比上次更好 / 一样 / 更差 / 新问题”展开。
- 所有页面优先展示变化，而不是原始表单字段。

### 6.3 低负担

- 必填项必须少。
- 不确定的项目允许跳过。
- 可以分步提交，不要求用户一次填完全部模块。

### 6.4 结构化优先，备注兜底

- 关键指标必须结构化，方便长期趋势与比较。
- 自由文本继续保留，但只作为补充说明，不再承担主要分析功能。

### 6.5 不制造焦虑

- 首页不以“落后于他人”“分位较差”作为核心表达。
- 可视化应强调自我变化和管理建议，而不是人群排名。
- 风险表达要克制，避免无依据放大。

## 7. 新录入逻辑

系统应保留 `submission` 作为记录容器，但要把它定义为“一次基线建档 / 一次随访 / 一次事件记录”。

### 7.1 三类入口

- `基线建档`
  - 第一次使用时完成。
  - 用于建立疾病背景、当前总体状态与长期比较基线。
- `快速随访`
  - 日常最常用入口。
  - 目标是在 2 到 5 分钟内完成一次病情变化记录。
- `事件记录`
  - 用于出现新问题或关键事件时快速登记。
  - 例如跌倒、新增足下垂、开始使用 AFO、呼吸情况变化、上传新报告。

### 7.2 基线建档内容

基线建档应拆成 4 步，不要求全部字段都必填。

#### 第一步：基础信息

- 姓名或昵称
- 出生年份或年龄段
- 所在地区
- 确诊时间

#### 第二步：疾病背景

- 是否明确诊断 FSHD
- 如果知道，可选填写：
  - FSHD1 / FSHD2
  - D4Z4
  - 4qA / 4qB
  - 甲基化
- 家族中是否有人有类似情况
- 首发部位

#### 第三步：当前总体状态

- 是否可独立行走
- 是否容易抬手困难
- 是否存在面部无力
- 是否存在足下垂
- 是否存在气短或睡眠呼吸问题
- 当前是否使用以下辅具：
  - AFO
  - 手杖
  - 轮椅
  - 无创通气

#### 第四步：当前困扰

- 疲劳
- 疼痛
- 上楼
- 穿脱衣
- 梳头/抬手取物
- 走路稳定性

建议使用 0 到 4 或 0 到 5 的简单难度等级，并提供“跳过”。

### 7.3 快速随访内容

快速随访应围绕“变化”组织，建议按以下顺序：

#### 模块 1：今天和上次相比

- 总体感觉：更好 / 差不多 / 更差
- 是否有新问题
- 是否有一次重要事件需要记录

#### 模块 2：核心功能域

- 上肢：
  - 抬手是否更困难
  - 梳头 / 洗头 / 取高处物品是否受影响
- 下肢：
  - 走路是否更费力
  - 上楼是否更困难
  - 是否出现或加重足下垂
- 面部：
  - 闭眼 / 鼓腮 / 吹口哨 / 微笑是否有变化
- 呼吸：
  - 是否更容易气短
  - 平躺是否更不舒服
  - 是否影响睡眠

#### 模块 3：症状影响

- 疲劳评分
- 疼痛评分
- 跌倒次数
- 睡眠质量

#### 模块 4：干预与工具

- 是否开始或停止某种用药
- 是否开始使用或更换辅具
- 是否开始新的康复训练

#### 模块 5：报告与资料

- 上传 MRI / 基因 / 血检 / 肺功能等报告
- OCR 解析后先展示患者可理解的摘要
- 允许用户确认“摘要是否正确”

#### 模块 6：提交前预览

- 本次最主要变化
- 本次新增事件
- 本次上传资料
- 与上次相比的变化摘要

### 7.4 事件记录内容

事件记录是单独的快入口，适合 30 秒到 1 分钟完成。

建议支持以下事件类型：

- 跌倒
- 新增足下垂
- 新增抬手困难
- 新增呼吸不适
- 开始使用 AFO
- 开始使用轮椅
- 开始无创通气
- 上传新报告
- 其他自定义事件

每条事件至少包含：

- `eventType`
- `occurredAt`
- `severity`
- `description`
- 可选附件或报告引用

## 8. 针对 FSHD 的结构化录入建议

患者端不应强行要求非常专业的肌力分级，但应支持逐步升级。

### 8.1 首期必须支持的结构化域

- `左右分开` 的上肢、下肢关键功能
- 面部功能变化
- 疲劳
- 疼痛
- 跌倒
- 呼吸相关症状
- 日常生活影响
- 辅具使用
- 报告结构化摘要

### 8.2 建议支持的关键 metric

- `shoulder_abduction`
- `arm_raise_over_head`
- `elbow_flexion`
- `ankle_dorsiflexion`
- `knee_extension`
- `stair_climb`
- `ten_meter_walk`
- `sit_to_stand`
- `eye_closure`
- `lip_pursing`
- `fatigue_score`
- `pain_score`
- `dyspnea_score`
- `sleep_quality`
- `fall_count`

### 8.3 录入方式建议

- 对普通患者优先用“困难程度”“是否比上次更差”的表达。
- 对愿意细录的用户，再提供专业模式：
  - MRC 评分
  - 精确时间
  - 更细颗粒度左右侧指标
- OCR 导入的报告数据应标记为“系统解析”，不要和患者手动录入混淆。

## 9. 数据模型改造建议

不建议推翻现有模型。建议在当前 `patient_profiles`、`patient_submissions`、`patient_measurements`、`patient_function_tests`、`patient_activity_logs`、`patient_documents` 基础上做增强。

### 9.1 `patient_submissions`

新增字段：

- `submission_kind`
  - `baseline`
  - `followup`
  - `event`
- `summary`
  - 本次提交的人类可读摘要
- `changed_since_last`
  - 是否明确存在变化

### 9.2 `patient_measurements`

保留表，但增加以下字段：

- `side`
  - `left`
  - `right`
  - `bilateral`
  - `none`
- `metric_key`
  - 例如 `shoulder_abduction_mrc`
- `body_region`
  - 例如 `shoulder_girdle`
  - `upper_arm`
  - `face`
  - `ankle`
- `entry_mode`
  - `self_report`
  - `guided_assessment`
  - `ocr_import`
  - `clinician_entered`
- `device_used`
  - 辅具或测试设备

说明：

- 现有 `muscle_group` 可在过渡期保留。
- 新逻辑应逐步迁移到 `metric_key + side + body_region`。

### 9.3 `patient_function_tests`

当前表保留，并把楼梯测试从活动日志迁回这里。

建议首期支持：

- `stair_climb`
- `ten_meter_walk`
- `sit_to_stand`
- `timed_up_and_go`
- `custom`

建议补充字段：

- `side` 可空
- `protocol`
- `unit`
- `device_used`
- `assistance_required`

### 9.4 新增 `patient_symptom_scores`

用于存患者主观症状和日常负担，建议新增表：

- `id`
- `profile_id`
- `submission_id`
- `symptom_key`
  - `fatigue`
  - `pain`
  - `dyspnea`
  - `sleep_quality`
  - `anxiety_about_progression`
- `score`
- `scale_min`
- `scale_max`
- `notes`
- `recorded_at`

### 9.5 新增 `patient_daily_impacts`

用于记录患者真正感知到的功能影响：

- `adl_key`
  - `hair_washing`
  - `reaching_up`
  - `stairs`
  - `dressing`
  - `walking_outdoors`
- `difficulty_level`
- `needs_assistance`
- `notes`

### 9.6 新增 `patient_followup_events`

用于支持病程事件时间线：

- `event_type`
- `severity`
- `occurred_at`
- `resolved_at`
- `description`
- `linked_document_id`

### 9.7 `patient_activity_logs`

- 保留。
- 仅作为补充叙述和自由备注。
- 不再承载功能测试结果和关键结构化指标。

## 10. API 改造建议

### 10.1 提交模型

建议扩展：

- `POST /api/profiles/me/submissions`
  - 入参支持 `submissionKind`
- `GET /api/profiles/me/submissions`
  - 返回时包含 `submissionKind`、摘要、关键变化

### 10.2 基线建档

- `PUT /api/profiles/me/baseline`
- `GET /api/profiles/me/baseline`

如果不想新增路由，也可先通过现有 `profiles/me` 聚合返回，但聚合 DTO 里要明确区分基线信息和随访信息。

### 10.3 快速随访

新增或扩展以下资源：

- `POST /api/profiles/me/measurements`
- `POST /api/profiles/me/function-tests`
- `POST /api/profiles/me/symptom-scores`
- `POST /api/profiles/me/daily-impacts`
- `POST /api/profiles/me/followup-events`

### 10.4 病程摘要接口

建议新增：

- `GET /api/profiles/me/progression-summary`

返回内容应面向患者，而不是面向统计分析。建议包含：

- 当前总体状态
- 与上次相比的变化
- 关键变化列表
- 最近报告摘要
- 左右受累概况
- 推荐回顾项

### 10.5 趋势接口

现有 `GET /api/profiles/me/insights/muscle` 可保留，但需要调整产品用途：

- 不再把“全量人群分位”作为患者端主表达。
- 更优先返回：
  - 最近一次值
  - 与上次差值
  - 最近 3 到 6 次趋势
  - 是否明显下降

## 11. 前端信息架构改造建议

### 11.1 首页

首页不是档案总表，而是患者当前病程摘要页。

首页首屏应先回答 3 个问题：

- 我现在整体怎么样
- 和上次比有没有变化
- 最近最值得注意的是什么

首页建议结构：

1. 顶部状态卡
   - 今日状态
   - 最近随访时间
   - 是否有新变化
2. 变化摘要卡
   - 上肢
   - 下肢 / 步态
   - 面部
   - 呼吸
   - 疲劳 / 疼痛
3. 快捷入口
   - 快速随访
   - 事件记录
   - 上传报告
4. 最近事件
5. 最近报告摘要

### 11.2 数据录入页

现有 `p-data_entry` 不应再是一个巨大工作台。

建议改成入口页，先让患者选择：

- 基线建档
- 快速随访
- 事件记录

每个入口进入专属流程，而不是在一个长页面里混录所有类型数据。

### 11.3 档案页

档案页应作为“完整病程总册”，而不是原始记录列表。

建议结构：

1. 病例身份与疾病背景
2. 长期变化摘要
3. 左右体图
4. 多域趋势
5. 病程事件时间线
6. 历史报告与解析摘要

### 11.4 报告详情页

报告详情应面向患者解释，而不是只展示 OCR 字段。

建议结构：

1. 这份报告说明了什么
2. 和病程的哪一部分有关
3. 体图联动显示影响区域
4. 原始报告与结构化字段

## 12. 可视化设计规则

### 12.1 总原则

- 不要先展示原始数值表。
- 先展示结论，再展示证据。
- 不做复杂 BI 面板。
- 不把患者端做成医生判读界面。

### 12.2 体图

体图必须满足：

- 左右分开
- 正面 / 背面切换
- 颜色表示当前受累程度
- 描边或箭头表示相对上次变化

如果某区域无数据，显示“未记录”，不要伪造颜色。

### 12.3 趋势图

趋势图不建议只保留单肌群折线。

更适合患者端的方式是小倍图：

- 上肢变化
- 下肢 / 步态变化
- 面部变化
- 呼吸变化
- 疲劳 / 疼痛变化

每张图只回答：

- 当前值
- 和上次比
- 最近是否更差

### 12.4 时间线

时间线表达对象应从“提交记录”改成“病程事件”。

优先展示：

- 新增足下垂
- 跌倒增多
- 抬手更困难
- 使用新辅具
- 呼吸不适加重
- MRI / 基因 / 肺功能新报告

### 12.5 报告联动

- MRI 报告解析出的肌群或区域，应能联动到体图。
- 基因报告应输出患者能理解的结论卡，而不是仅罗列专业字段。
- 肺功能、血检等监测报告应进入对应功能模块，而不是只是存档。

## 13. 文案与交互要求

### 13.1 文案

- 优先用患者能理解的中文。
- 所有专业术语提供轻量解释。
- 所有问题尽量使用“你最近有没有感觉……”而不是“请录入某评分”。

### 13.2 交互

- 提供“跳过”“稍后再填”。
- 允许通过一键复用上次数据，只修改变化项。
- 每个步骤不超过 3 到 5 个核心问题。
- 上传报告后先给出一句患者版摘要，再允许查看专业详情。

### 13.3 状态

所有关键页面都必须有：

- 空态
- 加载态
- 错误态
- 无变化态

## 14. 实施优先级

### Phase 1

- 明确 `submission_kind`
- 把楼梯测试迁回 `function_tests`
- 增加左右侧能力
- 新增症状评分表
- 新增事件表

### Phase 2

- 重构 `p-data_entry` 为 3 个入口
- 实现快速随访流程
- 实现事件录入流程
- 首页改成病程摘要页

### Phase 3

- 体图左右分离
- 多域趋势图
- 事件时间线
- 报告与体图联动

## 15. 验收标准

完成后，系统应满足以下标准：

- 患者能在 2 到 5 分钟内完成一次快速随访。
- 患者无需理解复杂医学术语，也能知道自己“最近哪里变了”。
- 系统能明确表达左右差异和不对称受累。
- 功能测试、症状评分、病程事件、报告摘要都能进入同一条病程线。
- 首页展示的是“病情变化摘要”，不是“原始数据仓库”。
- 时间线展示的是“发生了什么”，不是“提交了什么”。
- OCR 结构化信息能进入患者可理解的摘要模块。
- 同群体分位不再作为患者端主反馈。

## 16. 给实现 Agent 的直接指令

你现在要改造一个现有的 FSHD 患者管理平台。这个平台主要服务对象是患者本人，目标是支持患者自录和长期随访，而不是构建医生工作站。你需要把现有“按报告、肌力、活动、用药堆表单”的录入逻辑，升级为“基线建档、快速随访、事件记录”三类面向患者任务的流程。整个系统要优先帮助患者回答“我最近有没有变化、变化在哪、要不要带着这些信息去和医生沟通”，而不是要求患者完成高门槛专业录入。

请严格遵守以下要求：

- 第一，所有设计必须以患者可理解、低负担、可长期坚持为优先标准。
- 第二，录入流程必须默认基于“与上次相比变化”展开，不要每次重填全量档案。
- 第三，必须支持 FSHD 的左右不对称特征，不能继续把同一肌群同时映射到左右两侧。
- 第四，功能测试、症状评分、病程事件和报告摘要要进入统一的病程线。
- 第五，不要把同群体分位、人群比较或复杂统计分析作为患者端主视觉。
- 第六，报告上传和 OCR 解析仍要保留，但它们是辅助信息源，不是患者主录入方式。

请按以下优先级推进：

1. 数据模型：保留现有 patient profile 体系，新增 submission_kind、laterality、symptom_scores、daily_impacts、followup_events 等支持长期随访的结构。
2. 接口层：把楼梯测试迁回 function tests，新增快速随访所需的结构化资源。
3. 前端录入：把 `p-data_entry` 改成三类入口，而不是一个混合大表单。
4. 前端展示：把首页和档案页改成“病程摘要 + 左右体图 + 多域趋势 + 事件时间线”。
5. 可视化：优先展示自我变化和当前状态，不以原始数值列表或人群分位作为主表达。

请输出和实现时都遵守以下验收标准：

- 患者 2 到 5 分钟能完成一次随访。
- 患者首屏能看懂最近变化。
- 系统能表达左右差异。
- 时间线能表达病程事件。
- 报告能被患者读懂，而不是只剩 OCR 字段堆叠。
