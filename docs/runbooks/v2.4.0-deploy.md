# v2.4.0 部署手册

> 第一份 production deploy runbook。覆盖 v2.3.1 → v2.4.0 升级路径，包含 staging 验证 checklist + prod 部署步骤 + 回滚命令。
>
> 与 prod 通用 release-checklist (`docs/release-checklist.md`) 配合使用：那份是通用规约，本手册补 v2.4.0 specific 的步骤。

## 0. 这次发了什么

v2.3.1 到 master 累了 **101 个 commit**：

- **主线功能**：完整 AI 患者问答 (KB 检索 + 三档隐私同意 + 移动端 SSE 流式回答 + 引用 + 调用历史 + 同意历史)
- **基建迁移**：医学知识库从 Chroma Cloud 迁到本地 pgvector
- **安全加固**：闭合 92 条审计 finding（27 原始 + 65 follow-up），覆盖路径穿越、跨用户隔离、PII 脱敏、LLM 取消、KB 服务认证、并发竞态

详情参见 `docs/releases/v2.4.0.md` (release notes)。

## 1. 部署前必备项

### 1.1 新增/必填 env 变量

`apps/api/src/config/env.ts#validateProductionEnv` 校验下列项；缺一项 prod 启动 fail-fast：

| 变量                   | 不允许的值               | 说明                                                                                              |
| ---------------------- | ------------------------ | ------------------------------------------------------------------------------------------------- |
| `JWT_SECRET`           | `change-me-super-secret` | 任意 ≥32 字符高熵随机串                                                                           |
| `OTP_HASH_SECRET`      | `change-me-otp-secret`   | 同上                                                                                              |
| `DATABASE_URL`         | dev 默认值               | prod Postgres 连接串                                                                              |
| `OTP_PROVIDER`         | `mock`                   | 必须设 `tencent` 且配腾讯 SMS 凭据                                                                |
| `CORS_ORIGIN`          | `*` 或空                 | 显式列移动 app 域名（多个用 `,` 分隔）                                                            |
| `OCR_PROVIDER`         | `mock`                   | 必须设 `embedded` 或 `baidu`                                                                      |
| **`KB_SERVICE_TOKEN`** | **空**                   | **本次新增。Python KB 服务的 bearer token**，至少 32 字符随机串；API ↔ KB 服务两边必须用同一个值 |

### 1.2 KB 服务必要变更

Python KB 服务的 `__main__` 在以下任一情形下没有 `KB_SERVICE_TOKEN` 就拒绝启动：

- `KB_SERVICE_HOST=0.0.0.0`
- `NODE_ENV=production`

也就是说，**容器 image 升级 + 环境配 token** 必须一起做。先把 token 注入 secret，再 rolling restart。

### 1.3 依赖

`apps/api/requirements.txt` 是否进了：

- `pgvector` (psycopg 适配)
- `sentence-transformers` (含 bge-m3 模型权重路径，~2.3GB)
- `python-docx` / `python-pptx` (KB 多格式解析)
- `pillow` / `pytesseract` / `pdf2image` / `paddleocr` (OCR + 图像 ingest)

prod 镜像构建必须包含上述 wheel + 模型预拉。

### 1.4 Postgres extension

prod 数据库需要 `vector` 扩展（pgvector）：

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

这一步 migration 006 会做，但前提是数据库 superuser 有权限装扩展。托管 PG（RDS、Cloud SQL）通常需要先在控制台启用 `vector` 扩展白名单。

## 2. Staging 验证 checklist

### 2.1 数据库迁移

#### Pre-flight — `schema_migrations` 里有 `*_down.sql` 行吗？

PR #58 之前，`migrate.ts` 在 forward 跑完 `011_status_check_constraints.sql` / `012_text_column_constraints.sql` 之后，会**紧接着把同名 `_down.sql` 当 forward 跑掉**，把刚建的 CHECK + trigger 全 DROP，并写两条 `*_down.sql` 行进 ledger。**任何在 PR #58 (commit `bbd403f`) 之前部署过 master 的环境（包括以前跑 staging 跑过的）都中招**。

执行 migrate 之前，先检查 + 恢复：

```sql
-- 1) 看 ledger 有没有 _down 行
SELECT id FROM schema_migrations WHERE id LIKE '%_down.sql';
-- 期待：(0 rows)
-- 如果有，就是 pre-#58 中招了，继续 step 2/3 修复

-- 2) 同时把对应的 forward 行也删掉（schema 里 CHECK / trigger 也是空，要重跑）
DELETE FROM schema_migrations
 WHERE id IN (
   '011_status_check_constraints.sql',
   '011_status_check_constraints_down.sql',
   '012_text_column_constraints.sql',
   '012_text_column_constraints_down.sql'
 );

-- 3) 重跑 forward（_down 已被 PR #58 的 _isForwardMigrationFile filter 排除）
-- 在 shell:
--   npm run db:migrate
-- 期待 stdout:
--   Applied 011_status_check_constraints.sql
--   Applied 012_text_column_constraints.sql
-- _down 文件不该再出现

-- 4) 验证 CHECK + trigger 真到位
SELECT COUNT(*) FROM pg_constraint
 WHERE contype='c' AND conname LIKE '%_check'; -- 期待 ≥ 22
SELECT tgname FROM pg_trigger
 WHERE tgname='patient_followup_events_same_profile'; -- 期待 1 行
```

跑完 pre-flight 再走下面的常规迁移。fresh prod 环境（从未跑过老 migrate.ts）不需要做 step 1-3，直接跑 forward。

#### 常规迁移

```bash
# 1. 备份 staging DB（迁移前永远先备份）
pg_dump $STAGING_DB > backup-pre-v2.4.0-$(date +%Y%m%d-%H%M).sql

# 2. 应用所有未跑过的迁移
npm run db:migrate

# 3. 对每个 NOT VALID 约束跑 VALIDATE CONSTRAINT 看历史脏数据
psql $STAGING_DB <<'EOF'
ALTER TABLE app_users               VALIDATE CONSTRAINT app_users_role_check;
ALTER TABLE auth_otps               VALIDATE CONSTRAINT auth_otps_status_check;
ALTER TABLE auth_otps               VALIDATE CONSTRAINT auth_otps_purpose_check;
ALTER TABLE otp_verification_codes  VALIDATE CONSTRAINT otp_verification_codes_scene_check;
ALTER TABLE patient_submissions     VALIDATE CONSTRAINT patient_submissions_kind_check;
ALTER TABLE patient_documents       VALIDATE CONSTRAINT patient_documents_status_check;
ALTER TABLE patient_documents       VALIDATE CONSTRAINT patient_documents_type_check;
ALTER TABLE patient_measurements    VALIDATE CONSTRAINT patient_measurements_side_check;
ALTER TABLE patient_measurements    VALIDATE CONSTRAINT patient_measurements_entry_mode_check;
ALTER TABLE patient_medications     VALIDATE CONSTRAINT patient_medications_status_check;
ALTER TABLE patient_followup_events VALIDATE CONSTRAINT patient_followup_events_severity_check;
ALTER TABLE ai_prompt_audit         VALIDATE CONSTRAINT ai_prompt_audit_consent_level_check;
ALTER TABLE ai_prompt_audit         VALIDATE CONSTRAINT ai_prompt_audit_redaction_mode_check;
ALTER TABLE ai_prompt_audit         VALIDATE CONSTRAINT ai_prompt_audit_status_check;
ALTER TABLE patient_activity_logs   VALIDATE CONSTRAINT patient_activity_logs_source_check;
ALTER TABLE patient_function_tests  VALIDATE CONSTRAINT patient_function_tests_type_check;
ALTER TABLE ai_consent_events       VALIDATE CONSTRAINT ai_consent_events_note_length_check;
EOF
```

**任何一条 VALIDATE 失败 = 历史数据违反约束**。处理路径二选一：

- 找出违例行，逐行修：
  ```sql
  SELECT id, document_type FROM patient_documents WHERE document_type NOT IN
    ('mri','genetic_report','blood_panel','other');
  -- 用 UPDATE 把异常值统一映射到 'other' 或正确值
  ```
- 留下 NOT VALID 永久（不调 VALIDATE）：CHECK 还会管新写入，旧数据不动

#### Pre-flight backfill — most likely culprits

历史数据上**最可能违反新 CHECK 的两列**：

- **`patient_documents.document_type`** — v2.3.1 时代直接把 OCR 子类型（`muscle_mri`, `biochemistry`, `ecg`, `pulmonary_function` …）写进去，v2.4.0 的 CHECK 只允许 4 个 canonical 值。控制器现在用 `canonicalizeDocumentType` 映射写入，但**老行**还在。
- **`patient_documents.status`** — 老 status 可能有 `pending` / `queued` 等不在 011 词表里的值。

跑 VALIDATE 之前先看一眼，决定 back-fill 还是留 NOT VALID：

```sql
-- document_type 离群值统计
SELECT document_type, COUNT(*) AS n FROM patient_documents
WHERE document_type NOT IN ('mri','genetic_report','blood_panel','other')
GROUP BY document_type ORDER BY n DESC;

-- status 离群值统计
SELECT status, COUNT(*) AS n FROM patient_documents
WHERE status NOT IN ('uploaded','processing','processed','parsed','needs_review','failed','parse_failed')
GROUP BY status ORDER BY n DESC;
```

建议的 back-fill UPDATE（按 PR #50 的 `DOCUMENT_TYPE_CANONICAL_MAP` 落 sub-type → canonical）：

```sql
-- document_type back-fill
UPDATE patient_documents SET document_type = 'mri'
 WHERE document_type IN ('muscle_mri');

UPDATE patient_documents SET document_type = 'blood_panel'
 WHERE document_type IN (
   'biochemistry','muscle_enzyme','blood_routine','thyroid_function',
   'coagulation','urinalysis','infection_screening','stool_test'
 );

UPDATE patient_documents SET document_type = 'other'
 WHERE document_type IN (
   'medical_summary','physical_exam','pulmonary_function',
   'diaphragm_ultrasound','ecg','echocardiography','abdominal_ultrasound'
 );

-- catch-all (any unmapped legacy sub-type → 'other')
UPDATE patient_documents SET document_type = 'other'
 WHERE document_type NOT IN ('mri','genetic_report','blood_panel','other');

-- status back-fill (legacy → canonical)
UPDATE patient_documents SET status = 'uploaded'  WHERE status IN ('pending','queued','');
UPDATE patient_documents SET status = 'parse_failed' WHERE status IN ('error','ocr_failed');
```

跑完 UPDATE 再跑 VALIDATE。

#### Trigger 单独验证

migration 011 加的 `patient_followup_events_same_profile` 触发器**不在 VALIDATE CONSTRAINT 体系里**（trigger 不是 CHECK）。验证它生效：

```sql
-- 跨 profile 引用应该被 trigger 拒绝
DO $$
DECLARE
  victim_profile uuid;
  attacker_doc uuid;
BEGIN
  SELECT id INTO victim_profile FROM patient_profiles LIMIT 1;
  SELECT pd.id INTO attacker_doc FROM patient_documents pd
    WHERE pd.profile_id <> victim_profile LIMIT 1;
  IF victim_profile IS NOT NULL AND attacker_doc IS NOT NULL THEN
    BEGIN
      INSERT INTO patient_followup_events
        (profile_id, event_type, occurred_at, linked_document_id)
        VALUES (victim_profile, 'other', NOW(), attacker_doc);
      RAISE EXCEPTION 'trigger did NOT block cross-profile linked_document_id';
    EXCEPTION WHEN raise_exception THEN
      RAISE NOTICE 'OK: trigger correctly blocked cross-profile linked_document_id';
    END;
  END IF;
END $$;
```

### 2.3 核心功能端到端

按从浅到深的顺序跑：

#### A. 注册 / 登录 / OTP

- [ ] 用 staging 手机号注册 → 期望 OTP 发送成功
- [ ] **同手机号几乎同时发两次 register** → 第一个 201，第二个 **409**（不是 500）
- [ ] 故意把腾讯 SMS key 改错，发 OTP → 期望 500；**再发一次正常 SMS key 仍能成功**（前一次失败没占用日限额）
- [ ] 5 次错密码连续登录 → 第 6 次返回 429 + retryAfter

#### B. 患者档案 + 文档上传

- [ ] 用户 A 上传一份 MRI 报告（OCR `classifiedType: 'muscle_mri'`）→ 期望 201 + DB 写入 `document_type = 'mri'`（不是 `muscle_mri` 触发 CHECK）
- [ ] 上传 `Content-Type: text/html` 的文件 → 通过 `GET /me/documents/<id>` 拉回时响应头 `Content-Type: application/octet-stream` + `Content-Disposition: attachment` + `X-Content-Type-Options: nosniff`
- [ ] 用户 A 创建 submission，用户 B 在自己的请求里塞 A 的 submissionId 调 `POST /me/measurements` → 期望 **404**
- [ ] 没有 profile 的新账号上传文档 → 期望 **404**（在 storage.save 之前）
- [ ] `curl 'http://api/api/profiles/me/documents/<id>?storageUri=local://../etc/passwd'` 路径穿越尝试 → 期望被拒

#### C. AI 问答（核心）

- [ ] 同意状态 `none`：移动端发 AI 问答 → 期望 **403** + `code: consent_required`
- [ ] 切到 `basic` 同意：发问 "FSHD 是什么" → 移动端看到 token 逐字流入，回答末尾 `[N]` 引用可点
- [ ] 切到 `precise` 同意：问 "我的 D4Z4 重复数怎么解读" → 答案能引用本人 D4Z4 数值，audit 行 `usedPersonalData=true`
- [ ] 问题问到一半把 app 切后台 → tail 后端日志，应看到 `res.on('close')` 触发 + SiliconFlow 调用 abort（不再继续打 token）

#### D. 同意 / 共享偏好

- [ ] 同一账号开 2 个 tab，几乎同时点 `dataDonation` 开关 → 第二个请求 `SELECT FOR UPDATE` 排队，最终两个写入互不丢失
- [ ] 切 `precise` 同意 → 同意历史页 (`/me/consent/history`) 出现新行；切回 `basic` → 又一行
- [ ] AI 审计页能看到所有调用，状态 `success` / `error` / `consent_denied` 三类都有

#### E. KB ingest（运维侧）

- [ ] 500MB 大 PDF 跑 `npm run kb:ingest:dry` → worker RSS 不超过 ~200MB
- [ ] 一个中文文件名（`档案.md`）在 macOS 创建 → push → Linux 上跑 `kb:ingest --prune` → 期望**不**被当 orphan 删除
- [ ] 故意构造一个含 `IGNORE ALL PREVIOUS INSTRUCTIONS` 的 chunk 跑 ingest → action log 出现 `injection markers detected` 警告（不阻断）

## 3. Prod 部署步骤

按部就班；任一步未通过不进下一步。

### 3.1 维护窗口决策

- migration 011 创建一个 `BEFORE INSERT/UPDATE` trigger，不会锁表
- 011 + 012 的 CHECK 都是 NOT VALID，应用瞬间完成
- **如果决定跑 VALIDATE CONSTRAINT**：会扫全表，建议放进维护窗口或低峰时段
- **KB 模型权重**：warm-cache rolling restart ~30s（`bge-m3` 已经在 `kb-model-cache` volume）。**首次 deploy 或 volume 重建**：从 HuggingFace 拉 ~2GB 权重，3-5 min 起步，慢网可能更久。docker-compose healthcheck budget 是 30 × 15s = 7.5 min；超时会被 orchestrator 误判 deploy 失败。强烈建议**部署前 pre-warm**：
  ```bash
  # 在生产宿主机或 deploy runner 上：
  docker compose pull kb-service
  docker compose run --rm --no-deps kb-service \
    python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('BAAI/bge-m3')"
  # 模型权重落到 kb-model-cache volume，正式 deploy 走的是 warm path
  ```
  pre-warm 后正式 deploy 的 KB 容器 healthcheck 在 30-60s 内就能 ready。

### 3.2 顺序

1. **PG 启用 vector extension**（如未启用）
2. **注入新 env**（特别是 `KB_SERVICE_TOKEN`）到 secret store
3. **备份 prod DB** (`pg_dump`)
4. **apps/kb 容器 image 升级 + rolling restart**（必须在 #2 之后，否则新 image 启动时找不到 `KB_SERVICE_TOKEN` 会直接退出 —— 见 `apps/api/knowledge_service.py` `__main__` 的 fail-fast check）
5. **apps/api 容器 image 升级 + rolling restart**（必须在 #4 之后 —— api 的 `/healthz/ready` 包含 `kbService.status === 'ok'` 闸门，kb 未 ready 时新 api pod 会一直返回 503，让 k8s/orchestrator 误判 deploy 失败回滚。`apps/api/Dockerfile` 的 `CMD` 在 `node dist/index.js` 之前会自动跑 `node dist/db/migrate.js`，所以 migration 003→012 跟随 api 启动一并完成，无需单独手动 step。如果想 migrate-first 而不 rely on auto-run，可在这一步之前手工 `npm run db:migrate` 一次。)
6. **可选 `VALIDATE CONSTRAINT`**（按维护窗口决定 —— 老数据 back-fill 跑过、监控显示稳定后再做）
7. **移动端 build 推 EAS / TestFlight / Play Store**
8. **盯监控 1 小时**（见 §3.3）
9. **24 小时复盘**

> ⚠️ Step 4 必须在 step 5 之前。早期 runbook 草稿把 api 排在 kb 前面，导致 api 重启后无限期等 kb ready 才能离开 503 状态。

### 3.3 上线后第一小时盯什么

- API `5xx` 率：基线 +50% 以上立刻准备回滚
- `/api/profiles/me/documents/upload` 失败率：canonicalize 漏的子类型会在这里炸
- `auth.register_failed` audit 行（`reason=exists_concurrent`）：突增 = 有人在猜手机号
- SiliconFlow API 调用量 + token 消耗：比 dev 估算高 2× 就关注
- KB service 容器 RSS：不应超过 4GB（bge-m3 加载后）

## 4. 回滚

### 4.1 Migration 回滚

```bash
# 011 / 012 都是 NOT VALID CHECK + 一个 trigger，回滚瞬间完成
psql $PROD_DB < db/migrations/012_text_column_constraints_down.sql
psql $PROD_DB < db/migrations/011_status_check_constraints_down.sql
```

两份 down script 都 `DROP CONSTRAINT IF EXISTS` / `DROP TRIGGER IF EXISTS`，可以重复跑。

### 4.2 镜像回滚

```bash
# kubectl 风格示例
kubectl rollout undo deployment/openrd-api --to-revision=<v2.3.1>
kubectl rollout undo deployment/openrd-kb  --to-revision=<v2.3.1>
```

镜像可以直接回滚 —— `KB_SERVICE_TOKEN` 留在 deployment env 里对 v2.3.1 无害（旧 image 不读这个 env，只是 ignore）。回滚后 **不要** 手工移除这个 env，否则后续 roll-forward 回 v2.4.0 时新 image 会因为缺 token 拒绝启动。

### 4.3 数据回滚

如果迁移期间已经有新写入（这次 release 的 status 词表新加了 `parsed` / `needs_review` / `parse_failed`），回滚 v2.3.1 后这些值会变成"未知 status"。两个选项：

- 接受不一致（v2.3.1 代码读到未知 status 默认按 `uploaded` 处理）
- 跑批量 UPDATE 把新 status 映射回旧 status：

```sql
UPDATE patient_documents SET status = 'processed' WHERE status = 'parsed';
UPDATE patient_documents SET status = 'failed'    WHERE status = 'parse_failed';
UPDATE patient_documents SET status = 'uploaded'  WHERE status = 'needs_review';
```

#### 上传文档（`api-uploads` volume）的备份策略

患者文档（PDF / 影像 / 基因报告等 PII）落地路径取决于 `STORAGE_PROVIDER`：

- **`local`**（docker-compose 默认）：写在 named volume `api-uploads` 即容器内 `/app/apps/api/uploads`。**`docker compose down -v` 或 image rebuild 会清空这个 volume**，等于永久丢失所有患者文档，没有 DR 路径
- **`minio`**（强烈推荐 prod / staging）：写在 MinIO bucket `MINIO_BUCKET_NAME`（默认 `medical-reports`），用对象存储自带的版本 / lifecycle / 备份机制管理

prod 部署强制走 `STORAGE_PROVIDER=minio` 路径，理由：

1. volume 不会跟容器生命周期挂钩，rebuild / 重调度安全
2. MinIO / S3 / OSS 都支持 cross-region replication + 版本化，符合医疗数据 DR 要求
3. `apps/api/src/config/env.ts:validateStorageEnv` 已经强制 prod 下不能用 `minioadmin` 默认凭据

如果迫不得已要在 prod 用 `local`：

- 每天凌晨备份 volume：`docker run --rm -v openrd_api-uploads:/uploads -v /backup:/backup alpine tar czf /backup/uploads-$(date +%F).tar.gz -C /uploads .`
- 备份上传到独立机器或对象存储
- `docker compose down -v` **绝对禁用**，只用 `docker compose down`（不带 `-v`）

### 4.4 Mobile 回滚

移动端无法"取消已发版"。如果新版本严重炸：

1. 立刻在 EAS / 应用商店把 v2.4.0 build 设为不可分发
2. 引导用户回退到 v2.3.1（如果商店还能装）
3. 已升级用户先盯监控判断是否需要紧急 hotfix v2.4.1

## 5. 已知关注点

- **NFC orphan 转换**：首次 `kb:ingest` 时 macOS NFD 形态的中文文件名会被 NFC 重新归一化，看起来像"全部重 ingest 一次"，是预期行为
- **chroma_cloud 退役**：现在每次 ingest 在 chroma_cloud 上会 warn `scoped stale-chunk delete unsupported`。如 prod 依然在用 chroma_cloud，老 chunks 不会被清。建议本 release 之后启动 chroma → pgvector 的搬迁
- **移动端缓存清理**：升级到 v2.4.0 后，老用户首次打开 app 会看到 QnA 历史被清（chat cache key 没变，但 logout 时被清的范围扩了；升级流程下大多数用户不会 logout，所以这条主要影响测试账号）
